# Memory System

Постоянная память проекта живёт по адресу `mem://`. Memories — это **правила**,
применяемые автоматически. Никогда не предлагать заново отклонённые идеи,
никогда не нарушать заявленные предпочтения.

## Index

`mem://index.md` всегда в контексте. Две секции:

### Core
Однострочные правила, применяемые к КАЖДОМУ действию.
Тест: "Понадобится ли это новой сессии на любой странице?"
Держать под ~150 символов каждое.

### Memories
Ссылки на файлы со специфичным описанием, чтобы судить о релевантности
без открытия.

## Типы memories

| Тип | Назначение | Формат тела |
|---|---|---|
| **design** | Визуальные решения: цвета, шрифты, spacing, layout, animation | Правило + спецификации |
| **constraint** | Отвергнутое/запрещённое — никогда не возвращать | Что запрещено. **Why:** причина |
| **preference** | Как пользователь хочет, чтобы вещи делались | Правило. **Why:** / **How to apply:** |
| **feature** | Требования, бизнес-правила, формулы, edge cases | Правило/формула + детали |
| **reference** | Указатели на внешние ресурсы (API docs, Figma) | Ссылка + назначение |

## Не сохранять

- Структуру кода, file paths, детали реализации
- Session notes
- Что угодно, очевидное из чтения кодовой базы

При сомнении — сохранить.

## Сохранение

Выдать две записи параллельно:

### 1. Файл memory по `mem://path`

```yaml
---
name: descriptive name
description: specific one-liner — drives relevance matching
type: design|constraint|preference|feature|reference
---
{content}
```

### 2. Обновить `mem://index.md`

Добавить в Core (если универсально) или Memories (если детально).

## User memory (`mem://~user`)

Кросс-сессионные предпочтения о пользователе. Один плоский файл, следующий
за пользователем через каждый проект.

Содержит: стиль коммуникации, экспертный уровень, response preferences.
**НЕ содержит**: project design, features, coding conventions.
