# Основные правила

## File Modification

- Редактирую содержимое файлов через инструменты модификации, не shell-командами
- Переименование, перемещение, удаление — через `mv`/`rm`
- Записи, правки и просмотры файлов выдаю как параллельные вызовы в одном ответе
- Смотрю файл, только если его содержимое неизвестно или устарело

## Parallel Command Execution

Можно запускать независимые shell-команды параллельно:
```bash
<cmd-1> & <cmd-2>
wait
```

Параллелим только команды, которые не конфликтуют:
- Нет общих файлов, каталогов, временных путей, артефактов
- Нет общего изменяемого состояния (node_modules, lockfiles, .git, кеши, БД, порты)
- Нет зависимостей между командами
- Нет объединённого вывода, который скрыл бы причины ошибок
- Background-команды требуют явной проверки готовности

## Error Recovery

- Если правка не удалась ("no match", "Failed to parse patch", "Invalid Context"):
  перечитать файл через `code--view` и повторить с обновлённым содержимым
- Если застрял на 3+ попытках одной ошибки: попробовать другой подход —
  упростить или разбить изменение на меньшие правки

## Ограничения

- Никогда не запускать сборку/typecheck вручную (npm run build, tsc) — это делает harness
- Никогда не использовать stateful git команды (add, commit, checkout, merge, rebase, etc.)
- Никогда не делать `find /` — ограничивать поиск каталогом проекта
- Никогда не использовать голый `sleep N` — опрашивать в цикле
- Никогда не отправлять shell-комментарии (# ...)
